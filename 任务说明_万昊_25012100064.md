## 二、任务背景与目标

Mixture of Experts（MoE）模型在推理时通常不会让所有专家同时参与计算，而是由路由器根据输入 token 的表示，为每个 token 选择少量 expert。由于不同请求的语义、任务类型、长度和上下文结构不同，各层被频繁命中的 expert 也会发生变化。

本任务希望利用已给出的 MoE 推理 trace 数据，学习 prompt 输入和 expert 命中分布之间的关系。在正式评测时，测试输入只包含 `request_id`、`input_ids`、`attention_mask`、`prompt_len` 等字段，不提供真实标签。模型需要为每条请求预测 48 个 MoE 层中 expert 的排序列表，评测方再基于隐藏真值计算多个 K 值下的 Recall@K 和 NDCG@K。

因此，本项目的目标不是预测单个 expert，而是对每一层的 128 个 expert 给出从高到低的排序，使真实热点 expert 尽可能出现在预测列表前部。

## 三、数据理解与处理

题目材料中提供了三类 trace 数据：

- `1k_chat_all.jsonl`：聊天类请求 trace，共 1000 条。
- `1k_code_all.jsonl`：代码类请求 trace，共 1000 条。
- `1k_math_all.jsonl`：数学类请求 trace，共 1000 条。

每条样本包含输入信息和 MoE 路由统计信息，核心字段如下：

- `request_id`：样本编号。
- `input_ids`：tokenizer 编码后的 prompt token 序列。
- `attention_mask`：有效 token 标记。
- `prompt_len`：prompt 长度。
- `task_family`：任务类别。
- `num_layers`：MoE 层数，本任务为 48。
- `num_experts`：每层 expert 数量，本任务为 128。
- `raw_counts`：形状为 `[48, 128]` 的专家命中次数矩阵。

数据处理流程如下：

1. 读取三类 jsonl trace 数据并合并。
2. 将 `input_ids` 和 `attention_mask` 统一截断或补齐到 `max_len=256`。
3. 将 `task_family` 转换为任务类别 id，作为额外输入特征。
4. 对每条样本的 `raw_counts` 按层归一化，得到每层 expert 的真实命中分布。
5. 按 85%/15% 划分训练集和验证集。

归一化后的标签保留了 expert 之间的相对热度，适合用于排序学习和分布拟合。

## 四、模型设计

本项目实现了一个轻量级 PyTorch 模型 `HotExpertPredictor`。模型输入为请求 token 序列，输出为 `[batch_size, 48, 128]` 的预测分数矩阵。

模型结构包括：

1. Token Embedding  
   将 `input_ids` 映射为 token 向量表示。

2. Masked Mean Pooling  
   使用 `attention_mask` 对有效 token 做平均池化，得到请求级语义表示。

3. 任务类型与长度特征  
   加入 `task_family` embedding 和 `prompt_len` 特征，使模型能够区分 chat、math、code 等不同输入模式。

4. 层感知表示  
   为 48 个 MoE 层分别设置 layer embedding。请求表示与 layer embedding 相加后，可以让模型为不同层学习不同的 expert 偏好。

5. Expert 输出头  
   对每一层输出 128 个 expert 的 logits，表示该层每个 expert 成为热点的置信度。

该设计的优点是结构清晰、训练速度快、依赖较少，同时能够显式建模“同一个请求在不同层有不同热点 expert”的任务特征。

## 五、训练方法

训练时将 `raw_counts` 按层归一化为概率分布，模型输出 logits 后使用 `log_softmax` 得到预测分布。损失函数采用 KLDivLoss，用于拟合真实专家命中分布。

主要训练配置如下：

| 配置项 | 数值 |
|---|---:|
| max_len | 256 |
| num_layers | 48 |
| num_experts | 128 |
| embedding 维度 | 64 |
| hidden 维度 | 128 |
| batch_size | 64 |
| epochs | 8 |
| learning_rate | 0.001 |
| weight_decay | 0.0001 |
| dropout | 0.15 |
| valid_ratio | 0.15 |

训练过程中根据验证集 `NDCG@16` 保存最优 checkpoint，最终模型保存在：

```text
outputs/best_model.pt
```

## 六、评价指标

本任务主要关注 Top-K 专家预测能力，因此使用 Recall@K 和 NDCG@K：

- Recall@K：衡量真实热点 expert 是否被预测列表前 K 覆盖，反映覆盖能力。
- NDCG@K：衡量预测排序质量，真实命中更多的 expert 排得越靠前，得分越高。

本地验证统计了 K=4、8、16、32 多个指标，方便观察模型在不同提交长度下的表现。

## 七、实验结果

在 3000 条 trace 样本上训练，按照 85%/15% 划分训练集和验证集。默认配置训练 8 轮后，验证集结果如下：

| 指标 | 数值 |
|---|---:|
| Recall@4 | 0.3033 |
| Recall@8 | 0.3793 |
| Recall@16 | 0.4649 |
| Recall@32 | 0.5788 |
| NDCG@4 | 0.5114 |
| NDCG@8 | 0.5406 |
| NDCG@16 | 0.5785 |
| NDCG@32 | 0.6271 |

从结果可以看到，随着 K 增大，Recall 明显提升，说明模型能够较稳定地覆盖一批真实热点 expert。NDCG 随 K 增大也持续提升，表明预测排序具有一定有效性。

## 八、预测与提交格式

正式评测时，使用测试输入文件生成提交结果：

```bash
python predict.py --input-jsonl "测试输入.jsonl" --checkpoint outputs\best_model.pt --output-jsonl "wh_0064.jsonl" --top-k 64
```

生成结果后可使用格式检查脚本：

```bash
python validate_submission.py --submission "wh_0064.jsonl" --min-k 16
```

提交文件每一行对应一条请求，格式如下：

```json
{
  "request_id": "req_000001",
  "predicted_experts": [
    [12, 5, 88, 3, 41, 7, 19, 0],
    [4, 17, 29, 0, 63, 8, 91, 2]
  ]
}
```

其中 `predicted_experts` 外层长度为 48，每一层是按置信度从高到低排列的 expert id 列表。expert id 范围为 0 到 127，同一层内不重复。

## 十、模型局限与改进方向

当前模型使用平均池化获得请求表示，优点是训练快、复现简单，但也存在一些限制：

- 没有显式建模 token 顺序，对复杂长上下文的语义捕捉不够充分。
- 训练数据规模只有 3000 条，模型泛化能力受限。
- chat、math、code 三类任务之间可能存在差异，当前只使用简单任务 embedding 表示。
- Top-4、Top-8 的精确排序仍有提升空间。

后续可以尝试以下改进：

- 使用 Transformer Encoder、CNN 或 attention pooling 代替平均池化。
- 为不同任务类别训练独立模型或使用更强的任务条件化结构。
- 对高命中 expert 设置更高损失权重，强化头部排序。
- 使用多模型集成，提高隐藏测试集上的稳定性。
- 增加交叉验证，减少单次划分带来的评估波动。
